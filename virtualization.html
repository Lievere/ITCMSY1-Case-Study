01 Approach Decision

Methodology of Virtualization and its Justification

We evaluated three different approaches to running software on servers. We considered stuff like: Do you give each employee a private room (Virtual Machines), an open-plan desk (Containers), or a mix of both depending on the role (Hybrid)? Each option has trade-offs.

WHY VIRTUALIZATION: Virtualization turns one physical server into many logical ones, each fully isolated from the others

Our Recommendation

We recommend a Hybrid Model critical services like user logins and databases get their own private, isolated environment (Virtual Machines), while lighter services like the company website and monitoring tools run in a more shared, efficient space (Containers). This is the industry standard approach for organizations of IT COMSEE Corp.'s size, and it keeps costs down without sacrificing security where it matters.

As noted by Tanenbaum & Bos (2015), hybrid virtualization allows organizations to match workload isolation levels to actual security and performance requirements, rather than applying a one-size-fits-all solution.

02 Hypervisor Selection

The Engine Behind Everything: Proxmox VE

A hypervisor is the software that makes virtualization possible it sits directly on the physical server hardware and creates and manages all the Virtual Machines running on top of it. Think of it as the building manager for a high-rise apartment block: it decides who gets which floor, makes sure no tenant disturbs another, and handles emergencies.

Quick Concept

There are two kinds of hypervisors. A Type 1 hypervisor runs directly on the hardware, like a foundation fast and efficient. A Type 2 runs on top of a regular operating system, like software inside Windows easier to set up but slower and unsuitable for production servers. We always use Type 1 for real infrastructure.

Limitations We're Being Upfront About

  • No phone support by default if something breaks at 3 AM, the free tier relies on community forums rather than a vendor helpline

  • Microsoft workloads running licensed Microsoft software in VMs requires additional configuration steps that are straightforward but not automatic

The Bottom Line

IT COMSEE Corp. is an 80–120 person organization not a global bank that requires guaranteed 15-minute vendor response times. The limitations above are real, but they are all manageable at this scale. More importantly, the money saved on VMware licenses (₱540k–₱1.35M per year) is money that stays in the hardware budget, giving the organization better servers instead of paying for a brand name.

Our recommendation: purchase the Basic Proxmox subscription for all three servers at approximately ₱18,000–₱36,000 per year total. This closes the biggest practical gap priority security patches at a fraction of what VMware would cost. If the company grows significantly in the future, migrating to a commercial platform is straightforward.

Three Servers, One Team

The three servers do not operate independently. They are connected into a single Proxmox Cluster meaning they are always aware of each other, always sharing information, and always ready to cover for one another. It is like a three-person relay team: if one runner stumbles, the next one grabs the baton and keeps going.

This cluster setup, managed through the Corosync cluster engine (Proxmox Server Solutions, 2024), gives the organization four critical capabilities:

  • ✔One Dashboard for Everything all three servers and all their virtual machines are managed from a single web page. No need to log into each server separately.
  • ✔Maintenance Without Downtime before rebooting a server for updates, its virtual machines can be silently moved to another server. Employees never notice.
  • ✔Automatic Recovery if a server fails unexpectedly, its virtual machines automatically restart on one of the other two servers within 30–60 seconds.
  • ✔Shared File Storage all three servers are connected to the same NAS (Network-Attached Storage) unit, so virtual machines can be moved between servers without copying any data.

04 Tools & Justification

The Software Stack What Each Tool Does

Six software tools form the backbone of this virtual infrastructure. Each one was chosen for a specific job. Hover any card for a plain-English explanation of why that tool is in this design and what problem it solves.

Proxmox VE 8.x
The Building Manager

Controls all virtual machines and containers across all three servers from one central web dashboard. The IT team's primary interface for everything.

Think of Proxmox as the property management software for an apartment building. It decides who gets which unit, handles move-ins, move-outs, and responds when something in a unit breaks down. (Proxmox Server Solutions, 2024)
KVM
Virtual Machine Engine

The technology inside Proxmox that creates fully isolated virtual machines. Used for login systems, databases, and file servers where privacy is critical.

KVM uses the server's CPU hardware directly to create a completely separate computer inside the physical one. It is the same technology used by Google Cloud and AWS for their cloud servers. (Red Hat, 2024)
LXC Containers
Lightweight Service Environments

Like a much lighter, faster version of a virtual machine. Used for monitoring tools, web proxies, and system utilities that don't need full isolation.

An LXC container starts in under a second and uses as little as 50 MB of memory. For comparison, a full virtual machine uses 1–2 GB just to boot. For low-risk services, this efficiency adds up quickly. (Linux Containers, 2024)
Docker + Compose
Application Packager

Packages each application service into a self-contained unit. Multiple services can be started, stopped, or updated individually without touching anything else.

Docker Compose lets the IT team describe an entire application stack in a single text file and start it with one command. It also makes it easy to roll back to a previous version if an update causes problems. (Docker Inc., 2024)
Proxmox Backup Server
Automatic Backup System

Runs as its own virtual machine on Server 2. Takes automatic snapshots of all other VMs and saves them to the NAS efficiently, without interrupting running services.

PBS only saves the parts of a VM that changed since the last backup like how cloud storage only uploads new photos. This makes daily backups fast and storage-efficient, and the backups can be locked so even ransomware cannot delete them. (Proxmox Server Solutions, 2024)
ZFS Storage
Self-Healing File System

The file system used on each server's local SSD drives. Automatically detects and corrects silent data corruption something conventional file systems cannot do.

ZFS treats data integrity as a first-class feature. Every block of data is checksummed. If a bit on disk silently flips (called bit rot), ZFS detects the mismatch and restores the correct data from the mirror copy automatically. No human intervention needed. (OpenZFS, 2024)

What About the Future? A Note on Kubernetes

Right now, Docker Compose handles the organization's containerized applications. This setup was designed to be directly upgradeable to K3s Kubernetes a lightweight version of the platform used by the world's largest technology companies to run thousands of services at scale.

No hardware changes are needed to make this upgrade. When IT COMSEE Corp. grows beyond 120 employees, the same three servers can run Kubernetes without modification the organization simply transitions its Docker Compose files to Kubernetes deployment files. This is a deliberate future-proofing decision baked into the current design. (Kubernetes, 2024)

05 Virtualization Security

Keeping the Virtual Infrastructure Secure

Running multiple services on the same physical hardware is powerful but it also means a security problem in one place can potentially affect others if proper safeguards are not in place. Think of it like a multi-tenant office building: if the lobby is not secured, everyone inside is at risk regardless of how good the individual office locks are.

According to NIST's Guide to Security for Full Virtualization Technologies (2011), each layer of a virtualized environment the physical hardware, the hypervisor, the virtual machines, and the guest operating systems must be independently hardened. We have addressed each layer below.

Protecting the Control Panel
Must Do

The Proxmox web dashboard is the most powerful access point in the system whoever controls it controls every virtual machine and all data on all three servers. If this dashboard were exposed to the public internet without protection, it would be the equivalent of leaving the master key to every office on the front doorstep.

Protection PlanThe dashboard (port 8006) is restricted so that only IT Support workstations on the internal management network can reach it. Multi-factor authentication is required for all admin logins. Security patches are applied within 48 hours of release using the Proxmox subscription repository. (Proxmox, 2024)
Preventing Services from Talking to Each Other
Must Do

By default, virtual machines on the same physical server can communicate with each other across the internal network bridge as if they were on the same Wi-Fi network. This means without deliberate separation, the company's payroll database could theoretically be reached from a web application running on the same server.

Protection PlanEach department (Administration, Operations, IT Support) is assigned its own VLAN a logically separate network. Proxmox's built-in firewall then enforces rules about which services are allowed to talk to which others, preventing unauthorized internal traffic. (CIS Benchmarks, 2023)
Container Isolation Risk
Important

Containers are efficient because they share the underlying operating system with the host server. The downside is that an unpatched vulnerability in the operating system kernel could, in theory, allow a compromised container to access the server itself. This is a known and well-documented risk in the industry.

Protection PlanAll sensitive services user logins, the company database, file storage run in full Virtual Machines, never containers. Containers are reserved for monitoring tools and low-risk web services. All containers run in "unprivileged" mode, which limits the damage any compromise could cause. (Linux Containers, 2024)
Backups That Can't Be Deleted
Important

Ransomware attacks increasingly target backup systems because if backups are destroyed or encrypted, the victim has no choice but to pay. A virtualized environment is particularly valuable to attackers because compromising the backup server gives access to all recovery points for all services simultaneously.

Protection PlanProxmox Backup Server uses immutable retention locks backups cannot be deleted before their retention period expires, even by an administrator. The backup server itself sits on a separate network credential set that is unreachable from the VM network. Recovery drills are scheduled quarterly to confirm backups actually work. (Proxmox Server Solutions, 2024)
No Shared Admin Passwords
Standard Practice

Many organizations run into a common problem: everyone in IT shares the same "root" or "admin" password. When something goes wrong, there is no way to know who made which change. And when a staff member leaves, the password has to be changed and communicated to everyone a process that frequently gets skipped.

Protection PlanEvery IT Support staff member gets their own named account in Proxmox with only the permissions they actually need. Accounts are linked to the Active Directory login system so when a staff member leaves and their AD account is disabled, access to Proxmox is automatically revoked as well. (CIS Benchmarks, 2023)
Keeping Admin Access Off the Main Network
Standard Practice

If the IT management interface the Proxmox dashboard and server SSH access shares the same network as regular employee computers, then any employee computer that gets infected with malware is potentially one step away from the most sensitive administrative controls in the organization.

Protection PlanAll server management (Proxmox dashboard, SSH) is placed on a dedicated management VLAN that is only reachable from physically designated IT Support workstations. SSH is configured to require cryptographic key authentication password-based login is disabled entirely.
Preventing One Service from Hogging Resources
Standard Practice

In a shared environment, one runaway service a database query that goes wrong, a backup job that starts at the wrong time can consume all available CPU and memory on a server, slowing down every other service for every employee at the same time. This is sometimes called a "noisy neighbor" problem.

Protection PlanEvery virtual machine and container is given a hard resource cap in Proxmox a maximum amount of CPU and memory it is allowed to use. The monitoring dashboard (Grafana) sends an alert if any service approaches its limit for more than five minutes, giving IT time to investigate before it becomes a problem.
Full Audit Trail of Every Action
Built In

Proxmox automatically logs every administrative action: who logged in, which virtual machine was created or deleted, which settings were changed, and when. This is the foundation of accountability if something goes wrong, IT can trace exactly what happened and when.

Enhancement PlanThese logs are forwarded to a centralized log server running in a dedicated container, where they are retained for 90 days and monitored for suspicious patterns for example, multiple failed login attempts or an admin account active at 3 AM on a Sunday. (NIST SP 800-125, 2011)

06 Advantages & Limitations

Pros and Cons of the Architecture

No technology is perfect. Below we walk through the biggest benefits of this design alongside the honest limitations and what the IT team does to manage each one. Click any item to read the full explanation.

✔ Getting More From Three Servers Than You Would From Ten
▾
The Benefit: Without virtualization, each physical server does one job a file server is always a file server, even at 2 AM when nobody is using it. With virtualization, all three servers share their resources as a single pool of 192 GB of memory and 48 processor threads. When the database is busy during the day, it automatically gets more resources. When backup jobs run at night, those resources shift to the backup service.

The Limitation: The shared NAS storage unit can only transfer data at 10 Gigabits per second total. If all three servers simultaneously try to do large data transfers for example, during a full backup combined with large file uploads they compete for the same bandwidth pipe.
How We Handle It: Hot data the stuff being actively used stays on each server's local SSD drives, which are blazing fast (3,500 MB/s). The NAS is used for shared files and backups, not for live database operations. This keeps the two types of traffic separate.
✔ If a Server Goes Down, Nobody Notices
▾
The Benefit: Proxmox's High Availability Manager monitors every virtual machine every few seconds. If a physical server fails power outage, hardware fault, anything every VM that was running on it is automatically restarted on one of the other two surviving servers within 30 to 60 seconds. Employees may see a brief slowdown at most; no data is lost and no manual IT intervention is required.

The Limitation: This automatic recovery requires the three servers to always be able to communicate with each other. If the private management network connecting them is disrupted, the cluster can enter a "split-brain" state where it cannot tell whether a server has failed or is simply unreachable and it may become cautious and stop making automatic decisions.
How We Handle It: The cluster heartbeat runs on a physically separate network cable and switch port from regular traffic. A "quorum device" a small persistent tie-breaker prevents split-brain scenarios from causing unnecessary disruption.
✔ Maintenance Without Any Downtime
▾
The Benefit: When a physical server needs a firmware update, a hardware replacement, or just a restart, the IT team does not have to schedule maintenance windows or warn employees. They simply move all running virtual machines to the other two servers while they are still running then safely restart the physical server. When it comes back up, VMs can be moved back. Employees experience nothing throughout this entire process.

What Makes This Possible: All three servers are connected to the same NAS storage unit. Because VMs store their disk files on this shared storage rather than on the local server, moving a VM between servers does not require copying any data it is nearly instant.
! This System Requires Trained IT Staff
▾
The Limitation: Proxmox, KVM, LXC, and Docker all have their own configuration nuances. An IT staff member without prior Linux server experience will need time to become productive with this stack. Incorrect VLAN or network bridge configuration in Proxmox can cause VMs to be unable to communicate a type of failure that can be hard to diagnose if you do not know where to look.

How We Handle It: One designated IT Support team member becomes the primary Proxmox administrator and undergoes structured training. Proxmox's web interface significantly reduces the complexity compared to managing raw KVM from the command line. All VM and container configurations are documented in a version-controlled repository so any team member can understand and reproduce the setup.
! Containers Are Not as Isolated as Virtual Machines
▾
The Limitation: This is a known trade-off in the industry, not a flaw unique to this design. Containers share the server's operating system kernel the deepest layer of the software. A serious, unpatched vulnerability in the kernel could theoretically allow a compromised container to access the host server or other containers running alongside it. This is why the security community treats container isolation as "good but not perfect."

How We Handle It: The most sensitive services user authentication, the company database, backups never run in containers. They are always inside full Virtual Machines with hardware-enforced isolation. Containers are strictly used for monitoring tools and low-risk web services. The kernel is kept patched on a regular maintenance schedule, minimizing the window of exposure. (NIST, 2011)

IT Staff Training Plan

T1 Proxmox VE Administration (Required for all IT Support staff)
▾
Duration: 3–5 days of structured learning + 1 week of supervised hands-on practice.

What it covers: Navigating the Proxmox web dashboard, creating and managing VMs, understanding VM states (running, stopped, paused, migrating), configuring storage pools, reading cluster health indicators, and performing basic troubleshooting when a VM goes offline.

Resources: Proxmox's official documentation (pve.proxmox.com), free YouTube training series by Lawrence Systems and Craft Computing, and the Proxmox community forum. The Proxmox subscription also includes access to community tickets answered by core developers.

Outcome: Every IT Support staff member can independently manage day-to-day VM operations without requiring specialist external support.
T2 Docker & Container Operations (Required for the designated admin)
▾
Duration: 2–3 days of structured learning + practical exercises deploying and updating a Docker Compose stack.

What it covers: Understanding the difference between images and containers, reading and modifying Docker Compose YAML files, starting and stopping services, viewing logs, updating a container to a new version, and recovering from a failed update using image rollback.

Resources: Docker's official "Get Started" documentation, Play With Docker (an interactive browser-based training environment), and the Docker Compose documentation.

Outcome: The designated Proxmox administrator can independently update, restart, and troubleshoot all containerized application services on Server 2.
T3 Backup & Disaster Recovery Procedures (All IT Support staff)
▾
Duration: 1 day of training + a mandatory live recovery drill within the first month of operation.

What it covers: How to verify that scheduled backups are completing successfully in the PBS dashboard, how to restore a single file from a VM backup, how to restore an entire VM from a backup point, how to trigger a manual HA failover for testing, and what to do in a ransomware scenario (isolating affected VMs, restoring from immutable backup points).

Outcome: In an actual emergency, no single IT staff member is a bottleneck. Any member of the IT Support team can execute a full VM restore without external assistance. This is validated by the disaster recovery drill.
T4 Monitoring Dashboard Usage (All IT Support staff)
▾
Duration: Half a day walkthrough with ongoing self-service usage.

What it covers: Reading the Grafana dashboard panels for CPU, memory, disk I/O, and network utilization; understanding what normal baseline values look like; interpreting alert notifications sent by Prometheus; and knowing when a metric warrants immediate action versus scheduled investigation.

Outcome: The IT Support team moves from a reactive posture (fixing problems after users report them) to a proactive one (spotting resource trends before they become outages). This directly supports the organization's 99.9% uptime target.
